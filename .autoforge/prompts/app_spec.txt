<project_specification>
  <project_name>pdfme ERP Edition</project_name>

  <overview>
    A forked, extended build of the open-source pdfme library tailored as the document design and report generation engine within a NestJS/Next.js ERP platform. It provides a WYSIWYG template designer for ERP documents (invoices, statements, purchase orders, delivery notes, credit notes) and operational reports (aged debtors, stock on hand, sales analysis), with multi-tenancy, PDF/A-3b compliance, an expression engine, and a comprehensive REST API designed for seamless drop-in integration with a host ERP application. All customisation lives in the UI layer and new packages — upstream core packages (@pdfme/generator, @pdfme/common) remain unmodified to preserve upstream compatibility.
  </overview>

  <core_features>
    <designer_ui_redesign>
      - Three-panel layout: left panel (Blocks/Fields/Assets/Pages tabs), canvas (A4/Letter/etc.), right properties panel
      - Toolbar: template name (inline editable), page size selector, undo/redo, zoom control (25%-200%), preview mode toggle, generate preview PDF, save draft (auto-save 30s), publish, export JSON, version history
      - Blocks tab: draggable block cards grouped by category (Content, Media, Data, Layout, ERP)
      - Fields tab: searchable categorised tree of ERP data fields, drag-to-canvas binding, example value substitution
      - Assets tab: uploaded images/fonts/brand assets, drag-to-canvas, upload from panel
      - Pages tab: thumbnail strip, drag to reorder, context menu (duplicate, delete, set cover)
      - Canvas: real-time template rendering, rulers (px/mm), snap-to-grid (configurable, default 5mm), snap-to-element alignment guides, multi-select (cmd/ctrl+click), arrow key nudge (1px, shift=10px), delete/backspace to remove, page boundary with shadow
      - Properties panel: context-sensitive — Position and Size, Typography, Data Binding, Appearance, Schema Options, Constraints, Visibility (output channel), Page Visibility (pageScope), Conditional Visibility (expression-based)
    </designer_ui_redesign>

    <erp_schema_plugins>
      - Line Items Table: dynamic table with configurable columns, automatic page break with header repeat, multi-row groups (primary + conditional sub-rows), alternating group shading, footer rows (subtotal/VAT/total), maxRowsPerPage (number or {first, middle, last}), colSpan support
      - Grouped Table: hierarchical report layouts with groupBy (up to 3 levels), group header/footer rows with subtotals, multi-level nesting, SUM scoped to current group
      - ERP Image: storage-resolved assets (logos, stamps) via FileStorageService
      - Signature Block: signature line with configurable label, optional sub-label from field binding
      - Drawn Signature: real drawn signature embedding from FileStorageService at render time, fallback behaviour (blank/placeholder/error)
      - Watermark: diagonal text overlay (DRAFT/COPY/VOID), configurable text/colour/opacity/rotation/font size, controlled by template variable
      - Calculated Field: expression-evaluated field with format string, evaluated at render time
      - Rich Text: limited HTML subset (bold, italic, underline, font size, colour, line breaks), lightweight WYSIWYG editor, field bindings within rich text
      - QR/Barcode: QR code with ERP URL binding
    </erp_schema_plugins>

    <field_schema_system>
      - FieldGroup and FieldDefinition TypeScript types
      - Searchable tree with drag-to-canvas
      - Field binding stores {{field.key}} in schema content
      - Example value substitution in preview mode
      - Field type determines compatible schema types (image fields → image elements only)
      - Host ERP provides fieldSchema prop to Designer component
      - Field schemas registered per template type via FieldSchemaRegistry
    </field_schema_system>

    <expression_engine>
      - Built on expr-eval (MIT) with custom extensions
      - Arithmetic: standard operators with field references
      - String functions: LEFT, RIGHT, MID, UPPER, LOWER, TRIM, CONCAT, LEN
      - Conditional: IF (nested support), AND, OR, NOT
      - Date functions: FORMAT, DATEDIFF, TODAY, YEAR, MONTH, DAY
      - Numeric/Currency: FORMAT, ROUND, ABS
      - Locale-aware: FORMAT_CURRENCY, FORMAT_DATE, FORMAT_NUMBER (use org locale config)
      - Sandboxed execution — no Node.js globals, no require(), no eval()
      - Type coercion: numbers before arithmetic; null/empty → 0 for arithmetic, "" for strings
      - Expression editor in Properties panel with field picker and test button
      - 100% branch coverage required on all functions
    </expression_engine>

    <template_management>
      - CRUD with versioning and tenant isolation
      - Save mode prompt: "Edit in place" (overwrite published) or "New version" (draft, previous stays active)
      - Version history: version number, status, saved by, saved at, restore action, max 50 versions retained
      - Fork workflow: fork system template into org namespace
      - Template validation at publish time: binding check, expression parse, column sum, font availability, required fields, page scope sanity, orphaned elements
      - Template import/export: self-contained TemplateExportPackage with embedded fonts and images
      - Pessimistic edit locking: 30-minute timeout, 5-minute heartbeat, read-only mode for locked templates, force-release for admins
      - Auto-save every 30 seconds
    </template_management>

    <render_pipeline>
      - Synchronous renderNow: fetch published template → resolve data via DataSource → validate template → resolve page scopes → resolve conditions → resolve output channel → generate PDF via pdfme → PDF/A-3b conversion via Ghostscript → validate via veraPDF → compute SHA-256 hash → store via FileStorageService → create GeneratedDocument record
      - Async queueRender: Bull queue job, status polling endpoint
      - Bulk queueBulk: batch tracking with RenderBatch record, SSE progress endpoint, webhook callback, onFailure (continue/abort), max 2000 entityIds per request
      - PDF merge for batch downloads via pdf-lib
      - Preview PDF generation with sample data (5/15/30 rows), auto-watermarked "PREVIEW — NOT A LEGAL DOCUMENT"
      - Output channel awareness: elements tagged email/print/both, channel filtering at render time
      - Page scope resolution: all/first/last/notFirst, elements conditionally included per page
      - Conditional visibility: data-driven show/hide via fieldNonEmpty or expression evaluation
      - Filter application order: resolvePageScopes → resolveConditions → resolveSchema (channel) → generate
    </render_pipeline>

    <page_scope_system>
      - pageScope values: all (default), first, last, notFirst
      - resolvePageScopes() generates per-page template variants
      - estimatePageCount() via maxRowsPerPage or overflow detection
      - Designer: page visibility segmented control in Properties panel
      - Designer: page simulator in toolbar (1/2/3 page toggle)
      - Canvas: out-of-scope elements at 50% opacity with scope badge
      - Fully orthogonal with outputChannel — applied independently in sequence
    </page_scope_system>

    <output_channel_awareness>
      - outputChannel values: both (default), email, print
      - Pre-printed stationery support: suppress email-only elements when printing
      - resolveSchema() filters elements by channel before generation
      - Channel badges on canvas elements in designer
      - Channel preview toggle in toolbar
      - System templates authored with stationery group (email-only) and content group (both)
      - Render endpoints require explicit channel parameter
    </output_channel_awareness>

    <conditional_visibility>
      - ElementCondition: type fieldNonEmpty (render if field is non-empty) or expression (render if truthy)
      - resolveConditions() applied after page scope, before output channel
      - Properties panel: "Always visible" (default) or "Condition" with expression editor
      - Canvas badge: lightning bolt icon on conditional elements
      - Preview mode evaluates conditions against example data
    </conditional_visibility>

    <signature_system>
      - SignatureManager component: HTML5 Canvas drawing surface, signature_pad library
      - Capture as SVG path → transparent-background PNG → stored via FileStorageService
      - UserSignature model: orgId, userId, filePath, capturedAt, revokedAt
      - Signature revocation endpoint
      - drawnSignature schema plugin: pdf renderer fetches PNG at render time
      - Security: private directory (0700 permissions), authenticated API access only
    </signature_system>

    <pdf_a_compliance>
      - PDF/A-3b (ISO 19005-3) mandatory for all generated documents
      - Ghostscript conversion: -dPDFA=3 -dPDFACompatibilityPolicy=1 -sColorConversionStrategy=sRGB
      - veraPDF validation: reject and surface errors on failure
      - All fonts fully embedded — no system font references
      - Only open-licence fonts: Inter, Noto Sans, IBM Plex Sans, Roboto for system templates
      - Font fsType validation on upload: only flags 0 (installable) and 4 (preview and print) accepted
      - XMP metadata block with dc:title, dc:creator, xmp:CreateDate, xmp:ModifyDate, xmp:CreatorTool, pdfaid:part=3, pdfaid:conformance=B
      - PDF/UA (ISO 14289) as optional per-org setting for accessibility tagging
    </pdf_a_compliance>

    <localisation>
      - LocaleConfig: locale (BCP-47), currency (code, symbol, position, separators, decimal places), date (short/long format), number (separators)
      - Stored per-org in ERP settings, passed as prop to Designer and render API
      - Locale-aware FORMAT_CURRENCY, FORMAT_DATE, FORMAT_NUMBER expression functions
      - Multi-currency support: document-level currency indicator, dual-currency display
    </localisation>

    <file_storage_service>
      - Abstract FileStorageService interface: write, read, exists, delete, list, stat, usage
      - Default LocalDiskStorageAdapter: configurable rootDir and tempDir
      - Directory structure: {orgId}/documents/, {orgId}/assets/, {orgId}/fonts/, {orgId}/signatures/, system/fonts/, {tempDir}/previews/, {tempDir}/backups/
      - Temp file auto-purge (configurable retention, default 60 minutes)
      - Org-level isolation enforced at service level
      - Future extensibility: S3, Azure Blob adapters without changing consuming code
    </file_storage_service>

    <font_management>
      - Tenant-uploaded fonts via FileStorageService
      - Browser Cache API with 24-hour TTL for designer client
      - In-memory LRU cache (max 50MB) for render service
      - System fonts bundled in application under /app/fonts/
      - Font fallback to Noto Sans on missing font
      - Font validation on upload: fsType, file format (.ttf, .otf, .woff2), size limit (10MB)
    </font_management>

    <audit_trail>
      - Append-only AuditLog table: orgId, entityType, entityId, action, userId, metadata, createdAt
      - Written on every state-changing action by TemplateService and RenderService
      - No UPDATE or DELETE operations on audit table
      - Optional inputSnapshot on GeneratedDocument for exact document re-creation
      - PDF tamper detection: SHA-256 hash computed before storage, verification endpoint
    </audit_trail>

    <render_resilience>
      - Missing field binding: use fallbackValue or empty string
      - Missing image: placeholder rectangle with "Image not found" text
      - Missing signature: configurable fallbackBehaviour (blank/placeholder/error)
      - Expression error: configurable (empty string, "#ERROR", or fail render)
      - DataSource error: fail render cleanly, record error
      - Ghostscript failure: fail render, store raw PDF with _non-pdfa suffix for debugging
      - Font not found: fall back to Noto Sans, log warning
      - Retry logic: file storage 3x exponential backoff, database 2x, Bull queue attempts:3
      - Dead-letter queue: pdf-generation-dlq for failed jobs after all retries
    </render_resilience>

    <rate_limiting>
      - POST /api/render/now: 60 requests/minute per tenant
      - POST /api/render/queue: 120 requests/minute per tenant
      - POST /api/render/bulk: 5 requests/hour per tenant
      - Bulk batch size: max 2000 entityIds per request
      - Queue concurrency: configurable per-tenant (default 5, max 20)
      - Storage quotas: configurable per-tenant (default 5GB documents, 500MB assets)
      - HTTP 429 with Retry-After header on exceed
    </rate_limiting>

    <text_overflow>
      - Overflow strategy per text element: clip (default), truncate (with "..."), shrinkToFit (min 6pt)
      - Designer: segmented control in Properties panel
      - Preview mode simulates overflow behaviour with example value
      - Yellow warning badge when example value would trigger overflow
      - Dynamic vertical reflow NOT supported in v1 (pdfme absolute positioning constraint)
    </text_overflow>

    <backup_restore>
      - Export: ZIP archive with all org templates, assets, fonts, signatures, locale config
      - Import: ZIP archive → new draft templates (never overwrite), deduplicated assets, user validation for signatures
      - Temporary storage with 1-hour auto-purge
      - Use cases: staging↔production migration, tenant onboarding, disaster recovery
    </backup_restore>

    <datasource_registry>
      - Each document/report type registers a DataSource implementation
      - DataSource interface: templateType (string), resolve(entityId, orgId, params?) → PdfmeInputs[]

        - status (string — draft | published | archived)

        - updatedAt (timestamp, auto-updated)

        - lockedBy (string, nullable — userId holding edit lock)

      </Template>

      <GeneratedDocument>
        - id (cuid, primary key)
        - orgId (string)
        - templateId (string — FK to Template.id)
        - templateVer (integer — snapshot of version used)
        - entityType (string — invoice | statement | etc.)
        - entityId (string — FK to source record in host ERP)
        - filePath (string — relative path within FileStorageService root)
        - pdfHash (string — SHA-256 hash for tamper detection)
        - status (string — queued | generating | done | failed)
        - outputChannel (string — email | print)
        - triggeredBy (string — userId who initiated render)
        - inputSnapshot (jsonb, nullable — serialised inputs for audit)
        - errorMessage (string, nullable — error details if failed)
        - createdAt (timestamp, default now)
      </GeneratedDocument>

      <UserSignature>
        - id (cuid, primary key)
        - orgId (string)
        - userId (string)
        - filePath (string — relative path to PNG in private storage)
        - capturedAt (timestamp, default now)
        - revokedAt (timestamp, nullable)
        - Unique: [orgId, userId]
        - Index: [orgId]
      </UserSignature>

      <RenderBatch>
        - id (cuid, primary key)
        - orgId (string)
        - templateType (string)
        - channel (string — email | print)
        - totalJobs (integer)
        - completedJobs (integer, default 0)
        - failedJobs (integer, default 0)
        - failedIds (text array — entityIds that failed)
        - status (string — running | completed | completedWithErrors | aborted)
        - onFailure (string — continue | abort)
        - notifyUrl (string, nullable — webhook callback URL)
        - createdAt (timestamp, default now)
        - completedAt (timestamp, nullable)
        - Index: [orgId, status]
      </RenderBatch>

      <AuditLog>
        - id (cuid, primary key)
        - orgId (string)
        - entityType (string — template | generatedDocument | signature)
        - entityId (string)
        - action (string — created | updated | published | archived | forked | rendered | deleted)
        - userId (string)
        - metadata (jsonb, nullable — action-specific details)
        - createdAt (timestamp, default now)
        - Index: [orgId, entityType, entityId]
        - Index: [orgId, createdAt]
        - APPEND-ONLY: no UPDATE or DELETE operations
      </AuditLog>
    </tables>
  </database_schema>

  <api_endpoints_summary>
    <template_endpoints>
      - GET    /api/pdfme/templates                   → list org + system templates (paginated, cursor-based)
      - POST   /api/pdfme/templates                   → create template (draft)
      - GET    /api/pdfme/templates/:id               → get template by ID (includes full schema JSON)
      - PUT    /api/pdfme/templates/:id/draft         → save draft schema (requires saveMode when published)
      - POST   /api/pdfme/templates/:id/publish       → publish draft (runs validation gate)
      - POST   /api/pdfme/templates/:id/fork          → fork into current org
      - GET    /api/pdfme/templates/:id/versions      → version history
      - POST   /api/pdfme/templates/:id/restore       → restore a historical version as new draft
      - DELETE /api/pdfme/templates/:id               → archive (soft delete)
      - GET    /api/pdfme/templates/system             → list system templates
      - GET    /api/pdfme/templates/system/:id         → get system template JSON
      - POST   /api/pdfme/templates/import             → import TemplateExportPackage JSON
      - POST   /api/pdfme/templates/:id/validate       → run validation suite without publishing
      - POST   /api/pdfme/templates/:id/preview        → generate preview PDF with sample data
      - POST   /api/pdfme/templates/:id/lock           → acquire/renew edit lock (heartbeat)
      - DELETE /api/pdfme/templates/:id/lock           → release edit lock
    </template_endpoints>

    <render_endpoints>
      - POST   /api/pdfme/render/now                   → synchronous render — returns when PDF is ready
      - POST   /api/pdfme/render/queue                 → async render — returns job ID immediately
      - POST   /api/pdfme/render/bulk                  → batch render — returns batch ID
      - GET    /api/pdfme/render/status/:jobId         → poll single job status
      - GET    /api/pdfme/render/batch/:batchId        → get batch aggregate status
      - GET    /api/pdfme/render/batch/:batchId/progress → SSE stream for real-time batch progress
      - POST   /api/pdfme/render/batch/:batchId/merge  → merge batch PDFs into single file
      - GET    /api/pdfme/render/download/:documentId  → stream PDF binary (authenticated)
      - GET    /api/pdfme/render/verify/:documentId    → verify PDF integrity (SHA-256 hash check)
      - GET    /api/pdfme/render/history               → query generated document history (paginated)
    </render_endpoints>

    <asset_endpoints>
      - POST   /api/pdfme/assets/upload                → upload image, font, or brand asset
      - GET    /api/pdfme/assets                       → list org assets (paginated)
      - GET    /api/pdfme/assets/:assetId              → download/stream asset file
      - DELETE /api/pdfme/assets/:assetId              → delete asset (with reference warning)
    </asset_endpoints>

    <signature_endpoints>
      - POST   /api/pdfme/signatures                   → upload drawn signature
      - GET    /api/pdfme/signatures/me                → get current user's active signature
      - GET    /api/pdfme/signatures/:id/preview       → stream signature image
